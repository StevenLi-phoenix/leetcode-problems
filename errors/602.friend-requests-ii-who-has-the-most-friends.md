# 602. Friend Requests II: Who Has the Most Friends — submit rejected (401/403)

**Status:** unsolved / blocked on LeetCode's judge, not on our solution.
**Date first observed:** 2026-07-07

## Symptom

`leetcode submit 602 -l mysql` consistently fails:

```
✖ LeetCode rejected the request (session expired or invalid).
  Run `leetcode login` to sign in, or set LEETCODE_SESSION / LEETCODE_CSRF_TOKEN.
```

The underlying HTTP response is 401/403, mapped by the CLI to this generic "session
expired" message (see `~/Codes/leetcode-cli/src/api/client.ts`, `restPost()`).

## What still works fine on the same session

- `leetcode whoami` — reports logged in (local config check only, doesn't hit network).
- `leetcode test 602 -l mysql` — runs and returns **Accepted** against the example
  cases every time.
- `leetcode submit` on a *different* problem (474, C++) — succeeded immediately,
  same session, no re-login in between.

## Diagnosis (n=3 submit attempts across 2 independently re-exported browser
cookie files, both sessions confirmed valid via successful `test` + a successful
`submit` on problem 474)

Ruled out:
- **Stale/expired cookie** — re-exported fresh `LEETCODE_SESSION` + `csrftoken`
  from the browser twice; `submit` failed identically both times while `test`
  succeeded with the exact same credentials.
- **CLI bug (test vs. submit auth divergence)** — read `client.ts`: `interpret`
  (test) and `submit` both go through the same `restPost()` helper with identical
  `authHeaders()` construction (session cookie + `x-csrftoken` + referer). No
  code-path difference between them.
- **Global submit-endpoint rate limit** — submitting a different problem (474)
  on the same session succeeded right after 602 kept failing, so `submit` itself
  is not globally blocked for this account.

Working hypothesis (not confirmed, small n): the judge for **this specific
problem's submit path** (a MySQL/"database" category question) is broken or
otherwise rejecting requests server-side — a "ghost problem" in the sense that
LeetCode's own submission infra for it doesn't behave normally, independent of
our solution's correctness (which passed `test` against all example cases).

## Our solution (passes `test`, never confirmed by `submit`)

```sql
SELECT id, COUNT(*) AS num
FROM (
    SELECT requester_id AS id FROM RequestAccepted
    UNION ALL
    SELECT accepter_id AS id FROM RequestAccepted
) AS all_friends
GROUP BY id
ORDER BY num DESC
LIMIT 1;
```

been submit-confirmed).
